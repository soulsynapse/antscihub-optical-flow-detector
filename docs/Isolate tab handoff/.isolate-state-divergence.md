# Isolate handoff divergence ledger

This file records differences between the oracle handoffs and the rewrite
checkout. It is not a general rewrite status report and it does not amend an
accepted handoff by itself.

Current rewrite snapshot reported back to the oracle:

```text
checkout reviewed: 01b256f
milestone 3 implementation: 5bffca7
review date: 2026-07-23
```

The rewrite should refresh this ledger before implementing a later handoff when
its current types, ownership, lifecycle, or supported behavior differ from that
handoff's assumptions. Superseded observations are consolidated here instead of
being preserved as competing instructions.

## Current forward constraints

### Working-window source

- The rewrite has a Qt-free `application.working_window` boundary with
  `WorkingWindowRequest`, `ResolvedWorkingWindow`, native `rgb24` batches,
  request-local synchronous streaming, explicit extent provenance, and final
  outcomes.
- `WorkingWindowRequest` contains registered asset identity and absolute
  half-open temporal bounds. It intentionally contains neither source
  dimensions nor spatial tuning.
- `ResolvedWorkingWindow` contains probed native width and height, but obtaining
  it opens a request-local media session. UI-only geometry changes must not open
  that source.
- `IsolateSession.snapshot_working_window_request()` is a pure temporal request
  snapshot. It is not a general Isolate settings controller.
- GUI display generations remain GUI lifecycle state. They must not become
  reusable scientific identity.
- Scientific streams must continue to use their own media session rather than
  the mutable player session.
- Decoded or delivered source frames are not automatically scientifically
  examined coverage. A scientific consumer must establish its own processed
  span and outcome.

### Working grid

- `ActiveAsset` now carries registered native `width`, `height`,
  `content_sha256`, and sidecar/media references, but it remains in a
  PyQt-bearing module. Headless grid code should receive primitive values and
  must not import `ActiveAsset`.
- Registered `ActiveAsset.width,height` are the rewrite's available no-I/O
  geometry for immediate controls and painting. They are recorded facts, not a
  fresh verification of current media bytes.
- A later scientific consumer must require its probed
  `ResolvedWorkingWindow.width,height` to equal the captured grid's native
  extent before combining pixels with cells.
- Grid intent remains a plain Qt-independent value owned by `IsolateTab`.
  `IsolateSession` continues to own temporal playback and display-media
  lifecycle only.
- The current player has one letterboxed `image_rect()` and no general overlay
  transform. Grid projection should map normalized working boundaries directly
  into that rectangle.
- A grid change must create no `WorkingWindowStream`, no `MediaSession`, no grid
  worker, and no request to the display decoder.
- Dense-grid suppression is presentation only. It must not change resolved
  geometry or silently increase block size.
- Spatial controls remain usable for one-frame assets and must not inherit the
  temporal loop controls' `can_loop` gate.

## Open product decision

The rewrite has no established persistence policy for downsample or block
intent. Before milestone 4 implementation, the user must choose one session-only
asset-switch behavior:

```text
retain
  keep requested downsample and block intent, then re-resolve them against the
  new asset's native extent

reset
  restore downsample 1.0 and block auto whenever the active asset changes
```

Neither choice authorizes sidecars, `QSettings`, or another persistence layer in
milestone 4.

## Problematic rewrite findings to route separately

These were reported by the rewrite during the milestone-4 review. They are not
caused by the grid handoff and should not be absorbed into grid or first-channel
code. They should be tracked and fixed through focused lifecycle, identity, UI,
or diagnostic work.

### High — stale display-decode errors can defeat the newest request

The successful decode path discards an obsolete result when a newer request is
pending, but the `SieveError` path reportedly emits failure without the same
pending-request check. `IsolateSession._decode_failed()` can then close the
current media session for an error belonging to an older seek. The Replicates
decoder reportedly has the same missing stale-error guard.

Required follow-up: attach sufficient request identity to failures or discard an
obsolete failure before publication. Add a regression in which seek A fails
after seek B has superseded it; B must remain current and runnable.

### High — decoder shutdown has a start/interrupt race

The GUI reportedly calls `stop()`, waits with a timeout, drops the thread
reference, and closes the media session without checking whether the wait
succeeded. A worker can pass the stop interrupt before its FFmpeg process exists
and then start that process afterward.

Required follow-up: make stop visible immediately before decoder creation,
serialize decoder start/interrupt ownership, treat timeout as cleanup failure,
and verify both the worker and FFmpeg process have stopped. Apply one lifecycle
rule to Replicates and Isolate.

### High scientific-integrity risk — lineage may trust a stale recorded hash

Derivation reportedly performs metadata verification but can publish the
sidecar's recorded `content_sha256` as parent lineage without hashing the bytes
actually decoded. Same-size, same-dimension replacement media could therefore
produce a child attributed to stale content identity.

Required follow-up: distinguish recorded identity from content-verified identity
at processing/derivation admission and in published provenance. Do not imply
that a stored SHA-256 was verified during the current run when it was not.

### Medium — first-time registration can block the GUI

Opening an unregistered video reportedly probes, hashes the complete file, and
writes its sidecar synchronously on the GUI thread.

Required follow-up: either require deliberate prior registration or make the
cost observable and cancellable. Do not disguise a potentially multi-gigabyte
hash as an ordinary instant open.

### Medium — capped previews can distort aspect ratio

The media preview reportedly rounds scaled height and then forces it even while
holding width fixed, including a pinned `18x14 -> 9x8` case. Raw `rgb24` does not
obviously require even height.

Required follow-up: confirm a real backend constraint. If none exists, preserve
aspect ratio and update the regression. Grid projection may align with the
painted rectangle, but must not be used to legitimize distortion.

### Medium — the supported media benchmark underreports provenance

The reported benchmark does not yet consistently distinguish recorded from
verified content identity, preserve detailed frame-count provenance, report the
package/backend version, or include all accepted source, rate, sample, and
environment fields in its human output.

Required follow-up: complete the accepted versioned result schema and human
report without changing the honest boundary that the diagnostic measures media
decode rather than GUI painting or scientific processing.

## Required refresh before milestone 5

Milestone 5 is the first real scientific GUI consumer. Before implementing it,
the rewrite must report:

- The accepted milestone-4 grid types, ownership, asset-switch policy, and
  resolved-dimension rounding.
- How a captured grid is compared with probed working-window dimensions.
- The current working-window batch and cancellation APIs.
- The smallest existing GUI worker/supersession seam, if any.
- The current Isolate player-position signal and right-hand placeholder layout.
- Whether `rgb24` conversion metadata is explicit enough to define reproducible
  encoded-luma intensity.
- Any change to the problematic findings above that materially affects safe
  worker or result publication.

That report should adapt names and placement to the rewrite. It must not weaken
the scientific or stale-result requirements of the milestone-5 handoff.
